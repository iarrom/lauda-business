## 0. Что такое Lauda

Lauda — SaaS для инфлюенс-маркетинга, который помогает компаниям увеличивать охваты с помощью AI на международном рынке. CEO — Iar Romanov.

Этот репозиторий — единственный источник правды о бизнесе. Стратегия, исследования, сегменты, конкуренты, продажи, продукт, найм, финансы — всё здесь.

## 1. Архитектура (три слоя)

```
raw/   — immutable сырые источники. LLM ТОЛЬКО ЧИТАЕТ.
wiki/  — LLM-generated синтезированное знание. LLM ВЛАДЕЕТ ЦЕЛИКОМ.
CLAUDE.md — этот файл, правила игры.
```

### Слой 1: `raw/`
- **Никогда не модифицировать.** Это исходники: статьи, транскрипты, скриншоты, отчёты.
- Подпапки: `interviews/`, `meetings/`, `competitors/`, `market-research/`, `customer-feedback/`, `strategy-docs/`, `product-feedback/`, `financial/`, `assets/`.
- Имя файла: `YYYY-MM-DD-короткое-описание.md` (или `.pdf`, `.png`, `.txt`).
- Если источник — веб-статья, добавляй YAML frontmatter с `url`, `author`, `date`, `clipped_at`.

### Слой 2: `wiki/`
- LLM создаёт, обновляет, переписывает страницы. Iar (человек) только читает и просит.
- Структура подпапок описана в `README.md`.
- Каждая страница — markdown с обязательным frontmatter (см. раздел 3).
- Связи между страницами — через `[[wiki-link]]` (Obsidian-style).

### Слой 3: этот файл (`CLAUDE.md`)
- Описывает конвенции, workflows, типы страниц.
- Iar и LLM-агент **со-эволюционируют** этот файл по мере роста wiki. Если правило перестаёт работать — поменяй его в этом файле.

## 2. Три операции

### 2.1. Ingest — `ingest <путь к raw файлу>`

Когда Iar просит "ingest <файл>":

1. **Прочитай** файл из `raw/`.
2. **Обсуди ключевые тейкэвэи** с Iar коротко (3–7 пунктов). Если тейкэвэй противоречит существующей wiki — явно отметь это.
3. **Спроси разрешение** на план обновлений ("Я планирую создать X, обновить Y и Z. Поехали?"). При батч-ингесте без супервизии — пропускай этот шаг, но логируй честно.
4. **Создай summary** в `wiki/sources/summary-<короткое-имя>.md`.
5. **Обнови `wiki/index.md`** — добавь запись в нужный раздел.
6. **Обнови все релевантные страницы** концептов, сегментов, конкурентов, клиентов. Один источник может тронуть 5–15 страниц — это норма.
7. **Если найдено противоречие** — НЕ переписывай старое молча. Добавь в страницу секцию `## ⚠ Contradictions` со ссылкой на старый и новый источник и пометь обе как требующие ревью Iar.
8. **Допиши запись в `wiki/log.md`** в формате (см. раздел 4).

### 2.2. Query — `query <вопрос>` или просто вопрос

Когда Iar задаёт вопрос про бизнес/продукт/рынок:

1. **Сначала** прочитай `wiki/index.md`, чтобы найти релевантные страницы.
2. Прочитай эти страницы.
3. Если нужно — спустись в `raw/sources/` или сам raw-источник для верификации цитаты/цифры.
4. Сформулируй ответ с цитатами вида `[[wiki-page]]` и при необходимости — точными ссылками на `raw/...`.
5. **В конце ответа спроси**: "Хочешь, чтобы я зафайлил это как страницу в wiki?" — если ответ нетривиальный и компаундит знание.
6. Если зафайлили — создаётся страница в `wiki/comparisons/`, `wiki/decisions/` или нужном разделе, обновляется `index.md`, добавляется запись в `log.md`.

### 2.3. Lint — `lint` (раз в неделю, и/или перед стратегическими сессиями)

Health-check wiki:

1. **Противоречия** между страницами.
2. **Orphan-страницы** без входящих ссылок.
3. **Концепты, упомянутые 3+ раз, без своей страницы.**
4. **Устаревшие утверждения** — страницы, которые не обновлялись >60 дней и могли устареть (особенно про конкурентов, рынок, метрики).
5. **Дыры в данных** — для каких сегментов/конкурентов мало источников.
6. **Предложения** — какие следующие интервью / исследования провести.

Выход — отчёт `wiki/sources/lint-YYYY-MM-DD.md` + краткий разговорный summary в чат.

## 3. Конвенции страниц wiki

Каждая страница `wiki/**/*.md` ОБЯЗАНА начинаться с YAML frontmatter:

```yaml
---
title: Название страницы
type: concept | segment | competitor | customer | product | sales-deal | source-summary | comparison | decision | overview
sources: [список raw-файлов или wiki-страниц, на которых страница основана]
related: [[wiki-page-1]], [[wiki-page-2]]
created: YYYY-MM-DD
updated: YYYY-MM-DD
confidence: high | medium | low
owner: llm | iar
status: active | archived | needs-review
---
```

После frontmatter — содержимое в markdown с заголовками `##`.

### Типы страниц

- **concept** (`wiki/concepts/`) — общий концепт: "PMF", "AJTBD", "юнит-экономика", "AI-генерация контента".
- **segment** (`wiki/segments/`) — клиентский сегмент по методологии Замесина. Структура: Core Job, контекст, триггер, точки А и Б, барьеры, размер, consideration set.
- **competitor** (`wiki/competitors/`) — страница конкурента: позиционирование, цены, фичи, сильные/слабые стороны, ссылки на источники.
- **customer** (`wiki/customers/`) — карточка реального клиента или интервьюируемого: компания, роль, контекст, история взаимодействия.
- **product** (`wiki/product/`) — продуктовое решение, фича, гипотеза, RAT.
- **sales-deal** (`wiki/sales/`) — активная сделка: контакты, стадия, барьеры, следующие шаги.
- **source-summary** (`wiki/sources/`) — сжатый пересказ одного raw-источника + что из него попало в wiki.
- **comparison** (`wiki/comparisons/`) — сравнение (мы vs конкурент, варианты решения).
- **decision** (`wiki/decisions/`) — ADR-style запись стратегического решения: контекст, варианты, выбор, последствия.
- **overview** — только один файл `wiki/overview.md`, высокоуровневая картина бизнеса.

## 4. Формат `wiki/log.md`

Append-only. Каждая запись:

```markdown
## [YYYY-MM-DD HH:MM] <ingest|query|lint|update> | <короткое название>

**Source:** `raw/.../file.md` (для ingest)
**Question:** "..." (для query)
**Pages created:** wiki/sources/summary-x.md
**Pages updated:** wiki/segments/y.md, wiki/competitors/z.md
**Notes:** короткие заметки о противоречиях, важных инсайтах, что требует ревью.
```

Это парсится через `grep "^## \[" log.md | tail -10`.

## 5. Формат `wiki/index.md`

Каталог с разделами по типам страниц. Каждая запись:

```
- [[page-name]] — однострочное описание (N источников, updated YYYY-MM-DD)
```

Обновляй на каждый ingest и каждый зафайленный query.

## 6. Тон, язык, стиль для LLM

- **Язык wiki — русский.** Английский — только для имён продуктов, технических терминов, цитат из источников.
- **Сначала вывод.** Каждая страница начинается с TL;DR (3–5 строк) после frontmatter.
- **Без воды.** Никаких "в этом разделе мы рассмотрим", "стоит отметить, что". Сразу факт.
- **Цитаты — обязательно с источником.** Если факт пришёл из `raw/interviews/2026-04-10-client-acme.md`, ставь точную ссылку. Если из wiki — `[[wiki-page]]`.
- **Цифры — с датой.** "Конверсия 3.2% (по данным от 2026-04-15)".
- **Уровень уверенности — честно.** Если в `confidence: low` — пиши прямо в тексте: "Это гипотеза, не валидировано на 30 интервью".
- **Никогда не выдумывай факты.** Если данных нет — напиши "TODO: данных нет, нужно провести исследование X" и зафайль в `wiki/concepts/open-questions.md`.

## 7. Что НЕ делать

- ❌ Не переписывай `raw/` файлы.
- ❌ Не молча перезаписывай существующие утверждения в wiki — всегда явно отмечай конфликт.
- ❌ Не оставляй страницы без frontmatter.
- ❌ Не используй эмодзи в wiki (только `⚠` для contradictions и `✅` для validated assumptions).
- ❌ Не файли в wiki приватные данные (пароли, токены, личные данные сотрудников/клиентов помимо имени и роли). Если такие данные пришли — в саму страницу пиши маску `[REDACTED]`, в raw храни как есть, но `.gitignore` должен исключить.
- ❌ Не делай батч-ингест >5 файлов без явного запроса Iar.

## 8. Эволюция этого файла

Если правило перестаёт работать или появилось новое — обнови этот файл. Сделай запись в `wiki/log.md`:

```
## [YYYY-MM-DD] update | CLAUDE.md schema evolution
Reason: ...
Changed: добавил тип страницы X, изменил workflow Y.
```

## 9. Связь с методологией Замесина (AJTBD)

В этом проекте используется методология **Ивана Замесина (Advanced Jobs To Be Done)** как основная продуктовая рамка. См. установленный плагин `zamesin-pm`. Когда обсуждаются:

- **сегменты** — оформляй их по AJTBD-структуре (Core Job, Big Job, контекст, триггер, точка А, точка Б, барьеры);
- **PMF** — проверяй по 4 критериям Замесина;
- **интервью** — используй формат AJTBD-интервью / switch-интервью / expert-интервью;
- **ценность** — формулируй через 11 принципов создания ценности.

Это снимает необходимость переизобретать каждый раз — есть готовая рамка.

## 10. Команды-ярлыки (для Iar)

| Команда | Что делает |
|---------|------------|
| `ingest <file>` | Запустить ingest-workflow на указанный файл из raw/ |
| `query <вопрос>` | Ответить на вопрос на основе wiki, предложить зафайлить |
| `lint` | Health-check wiki, выдать отчёт |
| `open <страница>` | Открыть конкретную страницу wiki |
| `status` | Краткая сводка: сколько страниц, последний ingest, что в работе |
